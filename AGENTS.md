# エージェント共通ルール

各ルールは「**トリガー**(いつ)→ 行動(何をする)→ 理由(1文)」の形。トリガーに合致する作業を始める前に、該当ルールを適用する。

## このファイルの書き方(追記する人へ)

- 書いてよいのは「モデルのデフォルト動作からの差分」だけ: チームの決定、判断基準、失敗の補正(gotcha)。一般的なツールの使い方・手順は書かない — モデルは既に知っており、毎ターンのコンテキスト消費と陳腐化だけが残る。
- 1項目 = 1ルール。トリガーで書き始め、理由は1文まで。仕組みの解説(how)は書かない。
- **追記する前に**: 既存項目を検索し、同じテーマがあれば新項目でなく既存項目を更新する。このファイルは毎ターン全文ロードされるので、重複は恒常的に二重課金される。

## コミットと push

以下の `origin/main` はリモートのデフォルトブランチを指す(名前が異なるプロジェクトでは読み替える)。

- 1変更 = 1論理コミット。
- **worktree・作業ブランチを作るとき**: 起点は必ず `origin/main`(リモート ref)。ローカルの `main` は未 push コミットを含むことがあり、PR の diff に混入する。既にデフォルトブランチ上で変更・コミットしてしまった場合は、現 HEAD から作業ブランチを切って退避し、ローカルのデフォルトブランチを origin に揃え直す。
- **worktree を作るとき**: 配置先は `../worktrees/<リポジトリ名>/<issue番号 or ブランチ名>` にする。リポジトリ外に置くことで、誤って本体の worktree や gitignore に混入するのを防ぐ。
- **push する前**(履歴書き換え(amend・rebase)をしたら、次の push の前に再確認): ベース汚染を確認する。`git fetch origin <base>` してから — (1) `git diff --name-only origin/<base>...HEAD` のファイル一覧に、この PR が触るはずのないパスがないか目視する。(2) `git cherry origin/<base> HEAD` に `-` で始まる行(ベースと patch 等価な重複コミット — 誤った rebase/reset がマージ済みの作業を再導入した印)がないか確認する。どちらかに該当したら rebase で取り除いてから push する。
- **stacked PR を rebase するとき**: 依存 PR がマージされたら、`git rebase --onto` の前に必ず `git fetch origin <base>`。古い base への rebase は依存のマージ済み変更を無言で落とす(rebase は自分の diff しか replay しない)。rebase 後、依存のシンボルが HEAD に残っていることを grep で確認して完了とする。
- **push するとき**: コミットを溜めて1回で push する。push ごとに CI とレビューが走るため。

## 長時間ジョブの監視

- **デプロイ・CI・データ移行などを起動したとき**: 終端状態(成功/失敗/タイムアウト)まで監視し、結果とエラー内容を報告して完了とする。「トリガーした」という報告で作業を終えない。
- **監視対象を選ぶとき**: 実際のジョブ機構(workflow run、パイプライン実行)を特定してその状態を見る。推測した下流の効果(デプロイ後のリソース状態など)を代わりに見ない — 別の原因でも変化し、誤判定する。
- **ポーリングループを書くとき**: break 条件に「終端状態」か「回数上限」を必ず含める。「特定の名前のチェックが現れたら」のような、永遠に満たされない可能性のある条件だけで待たない。
- **CI に赤いチェックを見つけたとき**: conclusion を確認してから対処する。`cancelled` は失敗ではない(連続イベントで concurrency group が古い run を止めた痕跡)。最新の run が成功していれば対処不要。

## GitHub 操作

- **PR body を更新するとき**: 最新の body を取得し、必要なセクションだけを差分編集する。他者・bot の編集や、人間が添付した画像・動画を上書きで消さない。
- **`#` + 数字を含むテキストを投稿するとき**: 裸の `#6` はその番号の PR/issue へ自動リンクされる。順序数は出力言語の序数表現(「6番」/ "No. 6")かインラインコード(`` `#6` ``)で書き、投稿前に裸の `#<数字>` の紛れ込みを確認する。
- **レビューの承認状況を判断するとき**: review の state(`APPROVED` / `CHANGES_REQUESTED` / `COMMENTED`)だけを根拠にする。本文の文言は根拠にしない(「approve 済み」と書かれた `COMMENTED` レビューが実在する)。レビュアーごとに最新の state が有効で、新しいコミットが push されたら古い `APPROVED` は stale。
- **マージ済みかを判断するとき**: fetch 済みの `origin/main`(`git grep <symbol> origin/main` 等)と issue tracker / PR の state を根拠にする。ドキュメント内の status 表記とローカルの main checkout は stale になり得るので根拠にしない。

## テストとコマンド実行

- **自分の変更後にテストが失敗したとき**: 原因は自分の変更として扱う(main は green が前提)。setup・ビルド順・生成ファイルの問題に見えても「既存の問題」と結論しない。flake を疑うときは1回だけ再実行し、それでも失敗するなら flake ではない。
- **pass/fail を判定するコマンド(ビルド・テスト)を実行するとき**: パイプでフィルタに繋がず、素で実行する。
  - ❌ `make test | tail -20` — exit code が tail のもの(常に 0)になり、失敗が成功と報告される
  - ✅ `make test` を素で実行し、出力が長すぎたときだけフィルタを検討する

## コードコメント

- **コメントを書く前のテスト**: 「次の読者は識別子と周辺コードから読み取れるか?」— Yes なら書かない(WHY であっても)。書くに値するのは、識別子が伝えられない非自明な制約・回避策・驚きだけ。
- **書くとき**: 1行で簡潔に。経緯・チケット参照・詳しい理由はコミットメッセージと PR 本文へ(そちらは詳細でよい)。
- 書かないパターン(いずれも実際に削除を求められた実例):
  - ❌ 将来の予告 — 「アラート送信は TICKET-123 で実装予定」
  - ❌ 使用箇所の説明 — 「この値は X.y に保存される」
  - ❌ 既存慣習との比較 — 「Entitlement と同じ規約」
  - ❌ ファイル外の文脈に依存する WHY — フェーズ名や未実装コンポーネントへの言及(単体で読めない)

## コンテキスト管理(サブエージェントを持つエージェント)

- **手を動かす粒度の作業(調査・grep・ファイル編集・テスト実行・ログ精査)が発生したとき**: サブエージェントに委任し、メインのセッションは方針決定・指示出し・結果のレビュー・進行管理に徹する。自明な小修正でも、迷ったら委任側に倒す — メインのコンテキストは最も希少なリソース。
- **サブエージェントに委任するとき**: 対象パス(worktree)、使うべきツール・ラッパー、報告してほしい形式をプロンプトに明示する。サブエージェントはルールも会話文脈も継承しない。

## フォローアップ管理

- **作業を「先送りする」と決めたとき**: その場でフォローアップ issue を作り、コード内にリンクを残して完了とする。issue のない先送りは静かに消失するのと同じ。
- **期日のあるフォローアップ(期限日に設定を戻す等)を作るとき**: 期日付き issue に加えて能動的なトリガー(スケジュール実行・リマインダー)を必ず対にし、対象コードに期日と issue 番号のコメントを残す。issue だけでは当日誰も見ない。トリガーの指示文は文脈ゼロで実行できる自己完結の内容にする。

## ガードレール(手段の暴走防止)

- **テスト・lint・型チェックが通らないとき**: 実装を直す。チェックの弱体化(assert 削除、テストの skip、期待値の書き換え、lint ルール無効化、型エラーの ignore)で green にしない — チェックは要求仕様の一部。仕様自体が変わったためにテストを変えるときは、その変更を明示して報告する。直せないなら「直せない」と報告する。
- **pre-commit フックが失敗したとき**: 直してから commit する。`--no-verify` で飛ばさない — フックはローカルの CI で、飛ばしても失敗が CI に移るだけ。
- **破壊的操作(`reset --hard`・force push・ブランチ/ファイル/データの削除・環境の作り直し)をしたくなったとき**: 「とりあえずきれいにする」ためには使わない。実行前に (1) 何が失われるか(未 push コミット、他人の作業)を列挙し、(2) その操作が症状でなく原因に対応していることを確認する — 既知の失敗にパターンマッチした症状でも、原因は別のことがある。
- **同じアプローチで3回失敗したとき**: 続けない。`skills/consult-advisor` を呼び出して別モデルからの視点を得てから、次のアプローチを決める。アドバイスを受けても方向性が定まらない場合は、状況と試したことをまとめてユーザーにエスカレーションする。
- **実装・調査で行き詰まったとき**: 自己完結しようとしすぎず、`skills/consult-advisor` を呼び出して外部の視点を得る。アドバイサーは意思決定せず、助言を与えるだけ — 最終判断と実行は自分が行う。
- **依頼されていない問題に気づいたとき**: その場で直さない(勝手なスコープ拡大は diff とレビューを汚す)。フォローアップ issue にして報告する(「フォローアップ管理」)。
- **結果を報告するとき**: 失敗は失敗と、スキップはスキップと書く。「たぶん動く」を「動作確認済み」と書かない。試していないことを完了と報告するのは、後工程での発覚をより高コストにするだけ。
- **不可逆・外部公開の操作(削除、force push、外部サービスへの投稿・送信、本番環境の変更)**: 明示の指示があるときだけ実行する。過去に承認があっても文脈が変わっていたら再確認する。

## 学習ループ

- **再利用可能なナレッジ(規約、gotcha、修正された誤り)を得たとき — 遅くとも PR がレビュー可能になった時点・レビューを終えた時点**: メモリに記録する。メモリの記録場所は導入先の AGENTS.md で定義する(未定義ならユーザーに確認する)。定期的な昇格の手順と判断基準は `skills/promote-learnings`。
