# ナレッジの昇格

学習ループ **record → promote → distribute → apply** の promote を担う: メモリに溜まったナレッジを共有ナレッジへ移し、マージによって全員・全エージェントに配布され、次の PR で同じ問題が事前に捕捉される状態を作る。

## 昇格判定(全部 Yes なら昇格)

1. PR・spec・作者をまたいで再発しそうか?
2. 個人の好みではなく、具体的な事例に裏付けられているか?
3. 実行可能なチェックとして書けるか?
4. 特定の実装詳細に過剰適合していないか?
5. 得られる利益が、レビュー・コンテキストの負担増を上回るか?

どれか No なら昇格しない。特に次はメモリに留めるか捨てる: 一度きりの判断、検証されていない仮説、恒久ポリシーの顔をした一時的ワークアラウンド、レビュアーを過剰にブロックさせるルール、個人の対話スタイルの好み、進行中の作業状態。

## 置き場所の判定(必要になる頻度で決める)

1. **その作業の最中にだけ必要か?**(手順の gotcha、特定フローだけが踏むデータの癖)→ 該当スキルの SKILL.md に追記。オンデマンドでロードされるため、常時ロードの場所より常に優先。該当スキルがまだ存在しなければ、そのドメインのスキルを新設してよい(AGENTS.md に置くより優先)。
2. **エージェントが常に守るべき横断的な規約か?** → AGENTS.md に追記。毎ターン全セッションのコンテキストを消費するので、そのコストに見合うものだけ。既存の項目・セクションがあれば新項目でなく既存を更新する。
3. 新しいファイルを作るのは、横断的で既存の置き場所がない場合のみ。ドメイン固有ならスキル側(1)を選ぶ。

## 書き直しの規約

メモリは「未来の自分が文脈を思い出す」ための文体(Why・経緯つき)。共有ナレッジは「これから行動するエージェント」のための文体に直す:

- トリガー(いつ)で書き始め、行動(何をする/しない)とポインタ(コマンド、ファイル)を示す。理由は1文まで。
- 削るもの: インシデントの経緯、日付入りの武勇伝、内部の仕組みの解説。
- 追記先の既存構造に合わせる(該当セクションの下へ。末尾のぶら下がりブロックにしない)。

## 手順

1. メモリ(導入先の AGENTS.md で定義された記録場所)の索引と全エントリを読む。場所が未定義ならユーザーに確認する。
2. 各エントリを **昇格 / 保持 / stale**(参照先が消えた・現状と矛盾・既にルール化済み)に分類する。
3. 昇格候補ごとに「置き場所 + 書き直し後の文面」をユーザーに提案し、確認を得てから書き込む。
4. 追記前に重複を確認する。既に書かれていれば昇格せず、メモリ側を冗長として扱う。
5. 昇格したメモリは削除し、索引からも消す。
6. stale エントリは一覧でユーザーに示し、確認を得てから消す(修復したい場合があるため黙って消さない)。
7. 変更をコミットし、チームのレビュー経路(PR)に載せて完了とする。マージされて初めて配布される — 直接編集して終わりにしない。

## ローカル設定の昇格(エージェントが許可リスト等を持つ場合)

個人ローカルの許可設定(例: Claude Code の `settings.local.json`)のうち、**読み取り専用かつマシン非依存**のエントリをチーム設定(`settings.json`)へ移す:

- **移す**: 状態を観測するだけのコマンド(log / show / diff / status 系、view / list 系、lint・テスト実行)、読み取り専用スクリプト。
- **残す**: 書き込み・状態変更するコマンド、個人のパスや UID を含むエントリ。
- **移す前に stale 検出**: エントリが参照するスクリプト・パスが実在するか確認する。消えているものは一覧でユーザーに確認してから削除する。
- **移した後**: 両ファイルに重複がないこと、両方とも valid であることを確認して完了とする。
