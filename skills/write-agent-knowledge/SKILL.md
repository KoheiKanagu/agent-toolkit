# エージェント向けナレッジの書き方

## 出力の契約

書き上がったナレッジは次を満たす: (1) 初見のエージェントが会話文脈なしで従える、(2) 適用すべき瞬間に想起される形をしている、(3) 書いた分のコンテキスト費用に見合う。最後の「検証」まで終えて完了とする。

## 何を書くか — 「モデルのデフォルト動作からの差分」だけ

書いてよいのは3種類:

1. **チームの決定**(規約・役割分担) — モデルには導出不可能
2. **判断基準・チェックリスト** — 一貫性・抜け漏れ防止が要件になるもの
3. **gotcha(失敗の補正)** — モデルの一般知識がデフォルトで間違えることの記録

書かないもの: 一般的なツールの使い方・手順(モデルが既に知っている)、仕組みの解説(how)、インシデントの経緯や日付入りの武勇伝。判定テスト: 「この段落を消したら、初見のエージェントは違う行動を取るか?」— No なら消す。

## どう書くか — 7原則

1. **トリガーで書き始める**: 「**〜するとき**: 行動。理由(1文)。」— エージェントは状況からルールを検索するので、トリガー先頭の文が適用場面での想起率を決める。トピック主導の文(「〜は重要である」)は読めても想起されない。
2. **1項目 = 1ルール**: 複合箇条書きは片方だけ適用される事故のもと。分割すれば参照・更新も1ルール単位でできる。
3. **理由は1文だけ添える**: 理由ゼロの禁止則は、表面条件が違う場面で「今回は該当しない」と誤って一般化される。1文を超える理由は how の解説になるので削る。
4. **誤解リスクの高いルールには ❌/✅ の実例**: モデルは記述より例を強く模倣する。ただしトークン費用がかかるので、実際に解釈が割れた実績のある項目に限定する。
5. **判断は二値テストか「迷ったら X」で書く**: 「適切に」「なるべく」「必要に応じて」を残さない。判断のブレは基準の曖昧さに比例する。
6. **完了条件を明示する**: 「〜して完了とする」。エージェントは作業を早く切り上げがち — トリガー(開始)だけでなく終端も文にする。
7. **一貫した語彙**: 同じ概念には同じ語を使い、初出で1回定義する。言い換えは別概念に見える。
8. **プロジェクト依存の値をハードコードしない**: パス・ブランチ名・取得手段は「導入先の AGENTS.md で定義する」と明示し、導入側 README の定義一覧に載せ、未定義時のフォールバック(代替コマンド、ユーザーに確認)まで書く。値が暗黙に前提されたままだと、別プロジェクトで骨格機能が実行不能になる。

## SKILL.md の構造(この順で)

1. **frontmatter**: `name` と `description` のみ(エージェント固有のフィールドは移植性を壊す)。description には起動条件と、**使わない場面**を必ず書く — 過剰起動はスキルの主要な事故モード。
2. **出力の契約**: 最終成果物の形(形式、severity 分類、スキーマ、報告項目)を最初に置く。エージェントはゴールの形を先に知っていると途中の判断がそこへ収束する。
3. **手順**: 順序が意味を持つときだけ番号付き。持たないなら箇条書き。
4. **判断基準**: トリガー形式のルール群。手順と混ぜない(手順は「流れ」、基準は「分岐」)。

## AGENTS.md に追記するとき

- **追記の前に**: 既存項目を検索し、同じテーマがあれば新項目でなく既存項目を更新する。常時ロードされるファイルの重複は毎ターン二重課金される。
- **置き場所を確認する**: その知識が「特定の作業の最中にだけ」必要なら、AGENTS.md でなく該当スキルの SKILL.md へ(オンデマンドロードが常に優先)。判定フローは `skills/promote-learnings`「置き場所の判定」。

## 検証(書き終えたら)

**この会話の文脈を持たないサブエージェントに読ませて、初見で伝わるかをテストする**。自分で読み返しても検証にならない — 書いた本人は文脈で欠落を無意識に補完する。2観点:

1. **曖昧さ検出**: 「二通りに解釈できる文、未定義語、実行不能な指示、矛盾を挙げよ」と指示する。
2. **シナリオ遂行**: そのナレッジが対象とする状況を2〜4個与え、「手順を書き出し、ドキュメントが答えをくれず推測で補った箇所を全部正直に挙げよ」と指示する。推測で補われた箇所 = ドキュメントの穴。

指摘を反映して完了とする。
