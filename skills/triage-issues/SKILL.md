# issue の実装可能性トリアージ

## 出力の契約

- issue 1件につき: トリアージラベル1つ + complexity ラベル1つ + issue コメント1つ。ラベルは GitHub issue のラベルとして、コメントは GitHub issue のコメントとして付与する。それ以外にどこにも書き込まない(`needs-triage` の行動も代行しない — 推奨として列挙するだけ)。
- このスキルで使うラベルはリポジトリにあらかじめ作成しておく必要がある。未作成の場合は [scripts/setup-labels.sh](scripts/setup-labels.sh) を実行して作成してから起動する。
- 実行終了時の報告: issue → ラベル → expected outcome の表、ラベル別件数、残りの未トリアージ件数、停止時点の使用量。

## ラベルの判定基準

| ラベル                 | 付ける条件                                                                                                                                                                                                              |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `in-triage`            | claim マーカー。処理開始前に付け、判定確定時に外す                                                                                                                                                                      |
| `ready`                | 次の3つすべてを満たす: (1) 到達可能な情報源から期待挙動・再現手順・受け入れ基準が具体的に得られる、(2) 該当コードが特定できる、(3) チーム規約上必要な紐付け(課題管理チケット等)が揃っている                             |
| `needs-triage`         | `ready` の条件のどれかが欠けていて、埋めるのに人の行動が要る(情報不足、未決のプロダクト判断、スコープ曖昧、紐付けの欠如)。コメントに必要な行動を列挙する。追加情報が得られれば、次回のトリアージで `ready` に昇格しうる |
| `stale`                | この issue 自体の作業が不要: 解決済み(該当 PR/コミットを引用)、obsolete、他 issue の重複。close は人間が行う                                                                                                            |
| `complexity:low`       | 機械的・局所的な修正                                                                                                                                                                                                    |
| `complexity:medium`    | 数ファイルに触れる標準的な機能・バグ                                                                                                                                                                                    |
| `complexity:high`      | 横断的または設計判断を含む                                                                                                                                                                                              |
| `complexity:very-high` | アーキテクチャレベルまたは高リスク                                                                                                                                                                                      |

**`ready` か `needs-triage` かで迷ったら、`needs-triage` を選ぶ。**

## 判定の手順(issue 1件ごと、この順で)

1. **既存のトリアージコメントを読む**: 前回 `needs-triage` とされた場合は、追加情報や対応が入っているか確認する。足りるようであれば `ready` へ昇格する。
2. **解決済み・obsolete でないか**: 最近のコード・PR を検索する。該当したら `stale`(根拠リンク必須)。
3. **重複でないか**: issue のキーワードで open/closed 両方を検索し、タイトルでなく実質で比較する。重複なら `stale` とし、正とする issue(古い方・記述が充実している方・トリアージ済みの方を優先)を明示する。
4. **実装可能性**: 上の表の条件で `ready` / `needs-triage` を判定する。

## 並列実行プロトコル

- 未トリアージ一覧は**シャッフルしてから**処理する(並列実行が同じ issue に収束しないように)。
- **claim してから処理する**: ラベルを付ける直前に最新のラベル状態を再取得し、トリアージ系ラベルが既にあればその issue をスキップして次へ。claim に成功した issue だけ処理する。
- **判定が出たら**: コメント投稿 → 判定ラベル付与 → claim ラベル除去をまとめて行う。
- **処理できなかった issue**(途中で予算切れ・エラー): claim を外して返す。放置された claim は誰も再訪しない。
- **stale claim を見つけたとき**(`in-triage` ラベルだけあり判定ラベルがない): 付与から1時間以上経過していれば死んだ実行の残骸 — 外して claim し直す。

## トリアージ記録の形式

記録は GitHub issue へのコメントとして投稿する。コメントは自然言語で書く。読者は issue を見返す人間と次に対応するエージェントの両方。含める内容:

- 判定したラベル(`in-triage` / `ready` / `needs-triage` / `stale`)
- 付与した complexity ラベル(`complexity:low` / `complexity:medium` / `complexity:high` / `complexity:very-high`)とその理由
- 判定の根拠: どの情報に基づき、どう結論したか
- 次のアクション: 実装着手に必要なもの、または人間に求める行動、またはクローズの根拠リンク

必要に応じて箇条書きやリンクを使う。機械可読マーカーは並列実行の重複防止に使う場合だけ `<!-- agent-triage/v1 -->` のような行を先頭に置く — ただしこれはプロジェクトの運用に委ねる。

## 予算ゲート(バッチごとに確認)

- 指定された使用量上限(API 利用量・レート制限等)に達したら、停止して報告する。使用量の取得手段は導入先の AGENTS.md で定義する — 未定義のまま起動されたら、処理件数の上限をユーザーに確認してそれで代替する。
- 使用量の取得に失敗したら1回だけリトライする。それでも失敗したら理由を報告して止まる — 無制限にループしない。
