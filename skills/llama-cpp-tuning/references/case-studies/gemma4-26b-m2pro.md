# ケーススタディ: Gemma 4 26B on MacBook Pro M2 Pro

## 概要

本ケーススタディは、llama.cpp で `google/gemma-4-26B-A4B-it-qat-q4_0-gguf` を動かした際のチューニング結果を記録したものです。コーディングエージェント用途を想定して検証しています。

## 検証環境

| 項目 | 値 |
|------|-----|
| マシン | MacBook Pro 14インチ |
| チップ | Apple M2 Pro |
| CPU | 10 physical cores |
| メモリ | 32 GB Unified Memory |
| OS | macOS |
| llama.cpp | build 9870 (ggml 0.15.3) |
| 検証日 | 2026-07-05 |

## モデル情報

- **リポジトリ**: `google/gemma-4-26B-A4B-it-qat-q4_0-gguf`
- **アーキテクチャ**: Gemma 4, **Mixture-of-Experts (MoE)**
- **総パラメータ数**: 25.2B
- **アクティブパラメータ数**: 3.8B（推論時に計算されるのはこれだけ）
- **Expert Count**: 8 active / 128 total + 1 shared
- **量子化**: Q4_0（QAT適用済み）
- **コンテキスト長（推奨）**: 16384 〜 32768

## 最適パラメータ

最終的に見つかった最適なパラメータは以下の通り。

### llama-server 推奨コマンド

同じ内容の起動スクリプトを [`scripts/start-llama-server-gemma4-26b.sh`](../../scripts/start-llama-server-gemma4-26b.sh) に用意している。

```bash
llama-server -hf google/gemma-4-26B-A4B-it-qat-q4_0-gguf \
  -t 10 \
  -fa auto \
  -b 2048 -ub 2048 \
  -ngl -1 \
  -c 16384 \
  -np 1 \
  --no-mmap \
  --port 8080 \
  --host 127.0.0.1
```

### パラメータ解説

| パラメータ | 値 | 理由 |
|-----------|-----|------|
| `-t` | 10 | M2 Proの物理コア数に合わせる |
| `-fa` | auto | Flash Attentionはモデル・コンテキスト依存で最適が変わるため、自動判定を使用 |
| `-b` | 2048 | 長文プロンプト処理で効果あり |
| `-ub` | 2048 | `-b` と同値に統一 |
| `-ngl` | -1 | メモリに収まる範囲で全レイヤーをGPU載せ |
| `-c` | 16384 | コーディング用途で現実的な初期値 |
| `-np` | 1 | シングルユーザー・エージェント用途では並列リクエスト不要 |
| `--no-mmap` | - | Apple Silicon Unified Memory では、モデル全体を物理メモリに読み込む方が高速 |

## 検証した内容

### 1. スレッド数の影響

`-t` を 4, 6, 8, 10 で比較。物理コア数に近い `-t 10` が最も安定して高性能。

### 2. バッチサイズの影響

`-b` と `-ub` を 512, 1024, 2048, 4096 でスイープ。

- `llama-bench` 単体では大きな差は出にくい
- `llama-server` で長文プロンプトを処理する際に、`-b 2048` が顕著に効く
- 4096 まで上げても大きな改善は見られず、メモリ消費が増える傾向

### 3. Flash Attention

`-fa auto`, `-fa on`, `-fa off` を比較。

- `auto` が最も安全で高性能
- 一部ケースでは `off` の方が速い場面もあり、モデル・コンテキスト依存
- サーバー用途では `auto` を推奨

### 4. GPUレイヤー

`-ngl -1`（全レイヤーGPU載せ）が最速。メモリに収まるため問題なし。

### 5. 長文プロンプト

`-p 2048,4096,8192` で計測。バッチサイズ 2048 の効果が長文で顕在化。

### 6. プロンプトキャッシュ

`cache_prompt: true` で2回目以降のプロンプト処理が高速化。コーディングエージェントでは再利用が多いため有効。

### 7. KV キャッシュ量子化

`--cache-type-k q8_0 --cache-type-v q8_0` を試したが、Gemma 4 26B は層ごとに V の埋め込みサイズが異なるためエラーで利用不可。

### 8. mmap の有無

`--no-mmap` を試したところ、速度とメモリ使用量の両方で大きな違いが出た。

| 設定 | 物理メモリ使用量 | tg速度（短いプロンプト） | 備考 |
|------|---------------|------------------------|------|
| デフォルト（mmapあり） | 3GB未満 | ~36 t/s | ディスクI/Oがボトルネックに |
| `--no-mmap` | ~18GB | ~46 t/s | モデル全体を物理メモリに読み込む |

- mmapありではファイルをディスク上のまま仮想メモリ空間にマッピングするため、使用メモリは少なく見える
- Unified Memory 環境では、モデル全体をメモリに置いた方がディスクI/Oが発生せず高速
- メモリに余裕がある場合（32GB以上推奨）は `--no-mmap` を推奨

## llama-server 実測結果

### APIレイテンシ（参考値）

| プロンプト長 | 初回生成レイテンシ | キャッシュ後生成 |
|-------------|------------------|----------------|
| 短い（< 1K） | 低レイテンシ | ほぼ同様 |
| 中程度（2K） | バッチ2048で改善 | 改善顕著 |
| 長い（4K〜8K） | バッチ2048必須 | 大幅改善 |

※ 正確な数値は熱スロットリングや実行時の状態により変動するため、傾向として捉える。

## コーディングエージェント用途での所見

1. **プロンプトキャッシュは必須**: システムプロンプトやファイル内容の再利用が多いため、`cache_prompt: true` を活用する
2. **バッチサイズ 2048 を推奨**: 長いコードプロンプトを投げる際に効く
3. **コンテキスト 16K で十分**: 32K にしても利点よりメモリコストが大きい傾向
4. **シングルユーザー前提**: `-np 1` で十分。複数エージェント並列の場合は別途検証が必要

## 注意点

- **熱スロットリング**: 連続実行で性能が 10% 以上低下することがある。ベンチマーク間は 30〜60 秒冷却する
- **測定ばらつき**: 同じ設定でも ±10% 程度変動。大きな差（>20%）のみ判断材料とする
- **llama-bench と実APIの差**: `llama-bench` の数値は理想値。実際の `llama-server` レイテンシはオーバーヘッド込み
- **モデル依存**: 本結果は Gemma 4 26B の Q4_0 量子化モデル、M2 Pro 32GB 環境に固有。他のモデルや環境では最適値が変わる
