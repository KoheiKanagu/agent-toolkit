# PR レビュー

## 出力の契約

- 報告先はデフォルトで呼び出し元(会話・呼び出し元タスク)。GitHub へのレビュー投稿は明示的に指示されたときだけ行う — 投稿は取り消せない外部書き込み。
- 指摘は「ファイル・行参照 + severity + 検証済みの根拠」を1セットで出す。
- 問題がなければ「問題なし」と明確に言う。曖昧に濁さない。
- severity は4段階から必ず1つ:

| severity   | 基準                                                                       |
| ---------- | -------------------------------------------------------------------------- |
| must       | 修正が必須でバグ、セキュリティ、要件、仕様などに沿っていない。ブロッキング |
| should     | 修正した方が良いが、ブロッキングではない                                   |
| suggestion | 任意の改善。現状のままでもマージ可                                         |
| nit        | スタイル・可読性の些事                                                     |

## 手順(この順で)

1. **文脈**: 既存のレビュースレッドを読む — 未解決の懸念、作者の反論、他レビュアーの指摘を把握してから差分を読む。
2. **再レビューの場合の省略判定**: 自分が前回検証済みで、前回コミット以降変更のないファイルの主張は持ち越してよい。ただし次のどれかに該当したら再検証する — (a) 主張の根拠が別ファイルにあり、そのファイルが変わった、(b) 今回のラウンドで疑う理由が出た、(c) 時間依存の主張(外部の値、issue の現在の状態)。変更されたファイルは常に全検証。「前回のレビューが自分(同じ投稿アカウント)のもの」と確定できない場合は、この省略を使わず全件検証する。
3. **差分を読む**: 生成コード・lock ファイル・バイナリ・golden 画像はスキップ。差分だけで判断できないときは全文を読み、呼び出し元/先を辿る。
4. **5観点で評価**: 正確性(ロジック、エッジケース、null 安全)/ セキュリティ(injection、認可、情報漏えい)/ 性能(N+1、無駄なアロケーション)/ 保守性(命名、重複、関心の分離)/ テスト(変更を十分カバーしているか)。
5. **受け入れ基準と突き合わせる**: 基準は PR body → リンクされた issue / spec の順に探す。形式化された基準が見つからなければ、この手順は「PR body の宣言スコープと差分の整合確認」に縮退する — 基準を推測で創作しない。基準が見つかったら、基準ごとに「diff が満たすか」の表を作る。基準 = 拘束条件、実装方式 = 選択肢 — 方式の違いは基準を満たす限り指摘しない。満たさない項目は2択で分類する:
   - **PR の欠陥** — 基準が未達 → 指摘する
   - **spec の欠陥** — spec が固定した方式ではそもそも基準を満たせない → spec 作者へ差し戻す。PR のバグとして記録しない
   - issue に書かれた作業手順リストは計画であって拘束条件ではない。判断の軸はユーザー向けの挙動と PR body の宣言スコープ。
6. **出力**: 上の契約に従って報告する。

## 判断基準

- **チェック可能な主張は、チェックしてから指摘する。** 「非ブロッキングの注意」に格下げして検証を回避しない。「検証不能」と言えるのは実際に試して壁に当たった後だけで、何が壁だったかも書く。数学的・論理的な正しさ(座標変換、データフローの等価性)はコードを書いて実行して確かめる。
- **「ユーザー影響なし」という主張を見たとき**: 前提として受け取らず、主張を支える具体的なゲート(フラグ値、allowlist への登録、トリガー行を作る書き込みパスの不在)をコードかデータで確認する。確認できなければ、主張自体を未検証として指摘する。この主張は結論を支える柱 — 正しければ隣接リスク(マージ順、レース)も消えるが、崩れれば全体が崩れる。
- **「フォローアップは別 issue で」という記述を見たとき**: その issue が実際に作られてリンクされていることを確認する。なければ must。PR 自身のスコープに属するフォローアップは作者に作らせる(レビュアーが代作しない)。
- **「行を作成しない/永続化しない」という要件を見たとき**: 値のアサーション(`used == 0` 等)でなく、行数不変のアサーションがテストにあることを確認する。値のアサーションは将来 get-or-create に置き換わっても green のまま要件が破れる。
- **生成物が main の値を巻き戻しているように見えたとき**: must と判定する前に three-dot diff(`git diff base...HEAD`)と merge-base の内容を確認する。two-dot diff は「main が進んだ」を「ブランチが revert した」に見せる。ブランチが触っていなければ、必要なのは rebase であって修正ではない。
- **定量的な成功指標を見たとき**: それを計測する手段(ログ・集計)を用意する工程が存在するかを確認する。なければ、計測のフォローアップ issue を要求するか descope させる。計測できない指標は受け入れゲートにならない。

## spec / 設計ドキュメントの PR をレビューするとき

- 承認 = 開発のゴーサイン。承認後の spec は実装の契約なので、**should 以上はすべて must として扱う** — 修正させてから承認する。「承認 + コメント」で流してよいのは suggestion / nit だけ。
- **Why↔What の健全性を3点で確認する**:
  1. 課題は課題として述べられているか(解決策の先取りでないか)。テスト: 「これを作らなかったら誰の何が壊れるか?」への答えが弱ければ spec 全体が疑わしい。
  2. すべての要件が課題・期待成果にトレースするか。Why のない What はスコープクリープ、What のない Why は未達成のゴール。
  3. 要件は外から観測可能な挙動として書かれているか。合否判定できないほど抽象的なら受け入れ条件レベルまで下げさせる。特定のカラム・ウィジェット名まで指定しているなら非拘束の補足へ移させる。
- 要求する根拠の深さは決定のコストと不可逆性に比例させる: 安価で可逆な UI 変更は1文、データモデル・外部 I/F 変更は代替案 + 判断基準。
- 「現状維持」として引き継がれた要件にも Why を要求する。確立できないなら、基礎 spec のフォローアップ issue を作らせ、現 spec は「現在の挙動の表示」にスコープを絞らせる。
- spec が依存として挙げている issue は、書かれた時点でなく**現在の**状態を取得して確認する。spec は snapshot、issue は動く。
